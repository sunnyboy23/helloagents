# 方案设计模块

本模块定义方案构思和详细规划的执行规则，在 R2 标准流程下执行。

**核心职责:** 收集项目上下文（Phase1）+ 设计实现方案（Phase2），生成方案包（proposal.md + tasks.md）。

---

## 模块入口

```yaml
前置: EVALUATE 阶段完成（评估确认：核心维度全部充分 或 最后一轮追问确认合一已完成）
设置: CURRENT_STAGE = DESIGN
```

---

## 执行模式适配

按 G5 执行模式行为规范，本阶段补充规则:

### 模式行为

| 模式 | TASK_COMPLEXITY=complex | TASK_COMPLEXITY=simple/moderate |
|------|----------|----------|
| INTERACTIVE | 输出方案对比→用户选择→详细规划→进入 DEVELOP | Phase1+Phase2(跳过多方案对比)→进入 DEVELOP |
| DELEGATED | 推荐方案作为选择→详细规划→进入 DEVELOP | Phase1+Phase2(跳过多方案对比)→进入 DEVELOP |
| DELEGATED_PLAN | 推荐方案作为选择→详细规划→验收→结束 | Phase1+Phase2(跳过多方案对比)→验收→结束 |

⚠️ 选择方案 = 确认执行，不再有第二次确认

### 阶段切换

```yaml
方案类型约束:
  R2 标准流程（通用路径、~auto、~plan）: 类型始终为 implementation → 进入 DEVELOP
  overview 类型: 仅限 ~exec 入口（已有方案包为 overview 时）

方案确定后:
  overview 类型（仅 ~exec 入口）:
    INTERACTIVE → 询问归档 → 归档 → 输出完成 → 状态重置
    DELEGATED → 直接归档，总结中标注
  implementation 类型:
    设置 CREATED_PACKAGE = 方案包路径
    DELEGATED_PLAN → 返回 plan.md 执行验收
    其他 → CURRENT_STAGE = DEVELOP → 按 G7 加载开发实施阶段文件 → 进入 DEVELOP
```

### 推荐方案选择（DELEGATED模式）

```yaml
策略: 综合评估维度 → 优先"推荐"标记方案 → 无推荐则选用户价值与风险综合最优
记录: 在 proposal.md 中记录选择理由
```

## 执行流程

### Phase1: 上下文收集

> 按需加载: 按 G7 按需读取表"DESIGN Phase1 按需"行，根据条件加载所需文件

> 核心职责: 纯信息收集，不做设计决策。为 Phase2（方案构思）提供完整、准确的项目上下文。

#### 步骤1: 知识库开关检查

```yaml
KB_CREATE_MODE=0 → KB_SKIPPED=true
KB_CREATE_MODE=1 → KB_SKIPPED=false
KB_CREATE_MODE=2 → 编程任务→KB_SKIPPED=false | 非编程任务→KB_SKIPPED=true
KB_CREATE_MODE=3 → KB_SKIPPED=false
传递: Phase2 及 DEVELOP 阶段直接使用此值
```

#### 步骤2: 检查知识库状态

```yaml
前置: KB_SKIPPED=false 时执行

知识库状态显示:
  KB_CREATE_MODE=0: "⚠️ 已跳过（开关关闭）"
  存在: "✅ 已加载"
  MODE=1 不存在: "ℹ️ 不存在，已跳过（建议 ~init）"
  MODE=2 不存在: 编程任务→"🔄 已自动创建" | 非编程→同MODE=1
  MODE=3 不存在: "🔄 已自动创建"

MODE=1不存在 或 MODE=2非编程不存在:
  KB_SKIPPED=true
  继续执行
```

#### 步骤3: 初步复杂度评估

```yaml
评估: 按 G9 复杂度判定标准，基于已知信息初步设置 TASK_COMPLEXITY = simple | moderate | complex
依据: 需求描述中的工作单元数/涉及领域数/跨层级等维度（精确值待步骤5后确认）
脚本（可选）: project_stats.py

检测到 complex:
  输出: ℹ️ 复杂度: complex，已启用子代理编排
  继续执行
```

#### 步骤4: 获取项目上下文

```yaml
KB_SKIPPED=true → 扫描项目现有资源（代码库/文档/素材）
KB_SKIPPED=false → 知识库优先，不足则扫描项目现有资源

子代理调用（按需自动编排，遵循自动编排原则 [→ G10]）:
  现有项目资源 + ≥2个可独立扫描的目录/模块 → 按编排五步法调度子代理执行项目资源扫描 [→ G10 调用通道]
    编排: 按模块目录拆分，每个子代理负责扫描一个目录（子代理数=目录数，≤6/批）
    每个子代理 prompt 明确: 负责的目录路径 + 扫描内容（文件结构/入口点/导出接口/依赖关系）
    阻塞等待全部子代理返回后，主代理汇总各子代理实际返回内容，合并为完整项目上下文
  新建项目(目录为空或仅配置文件) 或 仅1个目录/模块 → 主代理直接执行
```

#### 步骤5: 提取关键目标与成功标准

```yaml
从完整需求提炼核心目标，明确可验证的成功标准
```

#### 步骤6: 深度分析与准备

```yaml
执行: 定位相关模块/素材 + 针对当前任务涉及的内容进行质量检查(过时信息/安全风险/一致性问题) + 问题诊断(日志/错误/冲突)
外部工具: 需要最新参考资料时按 G4 规则调用，保存/恢复 CURRENT_STAGE
输出物: 项目上下文（执行环境、项目结构、质量问题、约束条件）

子代理调用（按需自动编排，遵循自动编排原则 [→ G10]）:
  ≥2个可独立分析的单元 → 按编排五步法调度子代理执行深度依赖分析和质量评估 [→ G10 调用通道]
    编排: 按涉及单元拆分，每个子代理负责分析一个单元（子代理数=单元数，≤6/批）
    每个子代理 prompt 明确: 负责的单元路径 + 分析内容（依赖关系/接口/质量问题/风险）
    阻塞等待全部子代理返回后，主代理汇总各子代理实际返回内容，合并为完整技术分析报告
  仅1个单元 → 主代理直接执行

并行优化: 当步骤4和步骤6均需调用子代理且分析维度无交集时 → 并行调度 [→ G10 并行调度规则]
  否则 → 保持串行（步骤4先完成，结果传给步骤6）

复杂度确认: 根据完整分析结果修正 TASK_COMPLEXITY（若与步骤3初评不同则更新）

**DO NOT:** 做技术选型决策（留给 Phase2），设计实现方案（留给 Phase2），修改代码（留给 DEVELOP）
```

#### Phase1 完成

```yaml
输出: 上下文收集摘要（项目结构、关键目标、现有资源、约束条件）
继续: 直接进入方案构思与规划
```

### Phase2: 方案构思与规划

> 按需加载: 按 G7 按需读取表"DESIGN Phase2 按需"行加载所需文件

#### 步骤7: 模式分支判定

| 模式 | 特点 | 流程 | 子代理（概述，实际调用见各步骤） |
|------|------|------|--------|
| TASK_COMPLEXITY=simple/moderate | 跳过多方案对比 | → 步骤11 | 主代理直接填充方案包（步骤11） |
| TASK_COMPLEXITY=complex | 复杂任务多方案对比 | → 步骤8-13 | [RLM:brainstormer] 构思方案（步骤10）+ 主代理评估对比（步骤10）+ 主代理填充方案包（步骤11） |

#### 步骤8: 准备工作（仅 TASK_COMPLEXITY=complex）

```yaml
知识库检查: 使用 Phase1 已设置的 KB_SKIPPED 值
上下文获取: KB_SKIPPED=false→知识库读取 | KB_SKIPPED=true→扫描项目现有资源
项目规模判定: 使用 Phase1 已设置的 TASK_COMPLEXITY，影响任务拆分粒度
```

#### 步骤9: 分析判定（仅 TASK_COMPLEXITY=complex）

**项目场景判定:**

| 场景 | 特征 | 原则 | 填充深度 |
|------|------|------|----------|
| 新项目 | 初始化/从零开始 | 大胆创意，含需求视角分析 | 完整填充 |
| 现有项目 | 修改/优化/修复 | 精准执行，尊重现有成果 | 精简填充 |

**TASK_COMPLEXITY=complex = 多方案对比（强制，无例外）:**
TASK_COMPLEXITY=complex 任务的典型特征（多交付物/架构未定/技术选型/用户明确要求多方案）:
- 新项目初始化/重大重构/从零创建
- 架构决策/技术选型/方案选型
- 多种实现路径或内容组织方式
- 涉及多个模块/章节/领域
- 用户明确要求多方案

#### 步骤10: 方案构思与评估（仅 TASK_COMPLEXITY=complex）

⚠️ **串行强制:** 步骤10 整体与步骤11 之间串行:
步骤10: [RLM:brainstormer](各自独立构思方案，并行) → 主代理评估对比+标记推荐
步骤11: 主代理填充方案包
步骤10 与步骤11 之间有用户选择（INTERACTIVE）或自动选择（DELEGATED）

```yaml
TASK_COMPLEXITY=complex（CRITICAL）:
  方案构思: 调度 3~6 个 [RLM:brainstormer] 子代理（按需求复杂度决定数量），每个子代理独立构思一个差异化方案（并行）[→ G10 调用通道]
    - Codex CLI agent_type=brainstormer | Claude Code subagent_type="ha-brainstormer"
    - 全部 spawn 后立即阻塞等待返回（Codex CLI: collab wait，Claude Code: Task 自动阻塞），等待期间禁止执行任何其他步骤
    - 每个子代理接收相同的 Phase1 上下文 + 评估阶段确认的需求信息（含用户选择的成果规格偏好），独立输出方案（名称、核心思路、实现路径、成果设计（呈现方向/风格/体验）、用户价值、优缺点）
    - 子代理之间不共享中间结果，确保方案差异化（实现路径和成果设计方向均须差异化，不得仅路径不同而呈现方向趋同）。用户已在评估阶段确认的实施条件（如技术栈/工具/平台）各方案须遵守，差异化体现在架构模式/组件设计/内容组织/呈现策略等层面
    - UI 任务子代理额外约束: "方案须包含具体视觉设计方向（配色/布局/组件风格/动效），不能仅描述功能实现。视觉方向须差异化。"
  方案评估: 阻塞等待全部子代理返回后，主代理基于子代理实际返回内容统一评估对比 → 为最优方案标记"推荐"
    推荐标记: 推荐必须标记成果最完善、体验最好的选项，而非最简化最易实现的选项。推荐选项默认排序号1，（推荐）标记置于选项文本末尾
    评估维度: 用户价值（实用性/体验/现代使用场景）、方案合理性（专业性/可行性/可持续性/延展性）、风险(含EHRB)、实施成本（复杂度/工作量）
    权重: 按项目上下文动态调整（如性能关键系统提升方案合理性权重，原型/MVP 提升实施成本权重，品牌设计提升用户价值权重），用户价值始终不低于其他单项
    UI 任务权重修正: 交付物含 UI 时，"用户价值"中的"体验"子维度权重提升（视觉完成度、交互丰富度、布局专业度作为显式评估子维度参与加权）
    主代理基于完整上下文直接综合评估结果（不通过子代理中转）
  INTERACTIVE → 输出方案对比 → ⛔ END_TURN（等待用户选择方案N / 重新构思 / 取消(→状态重置)）
    用户选择后:
      方案N → 确认选择，进入步骤11
      重新构思 → 回到步骤10 重新调度子代理构思方案（最多重试 1 次，仍不满意则取消）
      取消 → 状态重置
  DELEGATED → 推荐方案直接进入步骤11

TASK_COMPLEXITY=simple/moderate: 直接确定唯一方案 → 步骤11
```

**INTERACTIVE 输出要素（按 G3 格式包装，场景=确认 ❓）:**
```yaml
主体内容:
  上下文收集摘要（项目结构、关键目标、现有资源、约束条件）
  （空行）
  方案对比（每个方案用 - 标记，非数字编号：名称、核心思路、成果设计方向、优缺点、推荐标记）
  （空行）
选项:
  选项：
  1~N. 各方案名称（推荐方案排序号1并标记推荐），N = 实际方案数
  N+1. 重新构思
  N+2. 取消
下一步引导: 引导用户回复选项编号，说明确认后进入详细规划并开始开发实施
```

**DO:** 所有方案详情完整输出后才提供选择

**DO NOT:** 在方案详情输出前提供选项；由主代理直接生成多个方案（必须通过 [RLM:brainstormer] 子代理并行构思，每个子代理独立输出一个方案）；在子代理未全部返回前执行后续步骤或生成方案内容（包括"先做其他事再回来看结果"后自行编造方案）

#### 步骤11: 详细规划

```yaml
脚本: create_package.py <feature> [--type <implementation|overview>]
目录创建: 按 G1 写入策略自动创建

脚本返回 success=false → 按 rules/tools.md AI降级接手流程处理

填充步骤:
  1. create_package.py 创建目录和模板
  2. 处理执行报告
  3. 主代理按 PackageService 接口规范直接填充 proposal.md（按场景深度）和 tasks.md（元数据头部+任务清单+状态符号+依赖声明）
     tasks.md 依赖声明: 每个任务项追加 depends_on 字段，声明前置依赖的任务编号 [→ G10 DAG 依赖调度]
     格式: [ ] 1.1 任务描述 | depends_on: [] 或 [ ] 1.2 任务描述 | depends_on: [1.1]
  4. 成果设计（含视觉产出的任务，填充 proposal.md "成果设计" 节）:
      设计思维（写代码前必须完成的创意决策，CRITICAL）:
        - 目的与受众: 谁在用？什么场景？什么情绪？
        - 美学基调: 选择鲜明方向，提交到一个明确的美学立场
          方向空间: 极简精致/大胆鲜艳/暗色科技/温暖有机/复古怀旧/游戏化活泼/
                    编辑杂志/粗犷原始/奢华精致/工业实用/像素复古/手绘插画...
          CRITICAL: 每个设计必须有明确的美学立场。大胆的最大化和精致的最小化都好，
                    关键是意图明确，而非强度。
        - 记忆点: 用户看到这个设计时会记住的一个特征
      视觉要素输出（写入 proposal.md "成果设计" 节）:
        - 配色: 大胆的主色+锐利的强调色优于胆怯的均匀分配，提交到一个有凝聚力的色彩体系
        - 字体: 有个性的展示字体(标题)+精致的正文字体配对，避免 Arial/Inter/Roboto 等通用字体
        - 布局: 非对称/重叠/对角线流动/破格元素/留白或密度控制 — 有意识的空间运用
        - 动效: 高影响时刻优先（一个精心编排的入场动画 > 散落的微交互），状态切换/滚动触发/交互反馈的动态效果
        - 氛围: 渐变/噪点纹理/几何图案/透明叠层/戏剧性阴影/装饰性边框/颗粒覆盖 — 创造纵深感而非默认纯色背景
      来源优先级:
        用户明确指定的视觉参考 > 步骤10子代理方案中用户选择的方向(TASK_COMPLEXITY=complex)
        > 从目的×受众×场景推导（推导时禁止使用"现代简洁"等空泛默认值）
      禁止: 通用AI美学（通用字体、紫色渐变白底、千篇一律卡片布局、
            无个性灰白配色、所有设计长得一样）
      跨次变异: 每次设计在明暗主题/字体/美学方向间变换，不同任务绝不趋同
      匹配原则: 实现复杂度须匹配美学愿景 — 大胆设计需要丰富代码，
               精致设计需要克制和精确
      创造力释放: 不要保守，突破常规，展现真正有设计感的作品
      非视觉任务: 跳过本步骤
  5. 技术约束（选填 proposal.md "技术约束" 部分）:
      可访问性/响应式/性能 — 仅列硬性约束，无则留空
  6. 设置 CREATED_PACKAGE
```

#### 步骤12: 方案包验收

```yaml
脚本: validate_package.py ${CREATED_PACKAGE}

验收项:
  ⛔ 阻断性: 结构完整(proposal.md+tasks.md 存在非空)，格式正确
  ⚠️ 警告性: 任务清单可执行，tasks.md 含元数据头部，方案覆盖评估阶段确认的成果规格

失败处理:
  INTERACTIVE → 输出: 警告（验证详情）→ ⛔ END_TURN（修复/强制继续/取消）
  DELEGATED:
    阻断性 → 中断委托 → ⛔ END_TURN（修复后重试/强制继续/取消(→状态重置)）
    警告性 → 记录继续
```

#### 步骤13: 输出与后续

```yaml
输出: 按执行模式适配规则
遗留方案包扫描: DELEGATED_PLAN 流程结束时执行
```

#### Phase2 完成

```yaml
输出: 方案包已创建（路径+类型+任务数）
继续: implementation → 进入 DEVELOP 阶段 | overview → 归档处理
```
